# Architecture

## Purpose and boundaries

The bot creates a small, private index for auto-service Telegram groups. A worker explicitly labels a message with `/car <plate> [note]`; the bot stores a normalized plate and a link back to that message. Search results never expose records from another configured chat.

The bot is deliberately not an OCR pipeline and does not retain image/video files.

## Runtime flow

```text
Telegram update
  -> explicit getUpdates loop (`src/polling.ts`)
  -> GrammY middleware and authorized handler (`src/index.ts`)
  -> parse / normalize / validate plate
  -> SQLite index (`src/database.ts`)
  -> reply with confirmation, search result, or inline keyboard
```

`runLongPolling` is intentionally explicit so only one `getUpdates` request is active. It requests both `message` and `callback_query` updates; the latter is required for `/list` buttons.

## Commands

| Command | Purpose | Scope |
| --- | --- | --- |
| `/car <plate> [note]` | Index a text message or photo/video caption. | Current allowed chat |
| `/find <plate>` | Return links to indexed messages for one plate. | Current chat only |
| `/list` | Show unique plates, newest mention first, ten per page. | Current chat only |
| `/start` | Show brief usage guidance. | Current allowed chat |

`/list` uses inline keyboard callback data such as `find:AA1234BB` and `list:2`. A selected plate deletes the list widget and returns the same response generated by `/find`.

## Data model

### `indexed_messages`

| Field | Meaning |
| --- | --- |
| `plate` | Validated, normalized plate token |
| `chat_id` | Telegram chat scope; mandatory for isolation |
| `message_url` | Direct Telegram message link |
| `message_preview` | Compact, normalized user note or fallback label |
| `media_type` | `photo` or `video` for new media records when known |
| `media_group_id` | Telegram album identifier, when applicable |
| `created_at` | Database creation timestamp |

Uniqueness is `(plate, chat_id, message_url)`: a message may be indexed once per plate per chat.

## Privacy and access model

- `ALLOWED_CHAT_IDS` is required at startup and is checked in handlers.
- All queries use the originating `chat_id`.
- The database contains no downloaded Telegram media and no full message/caption body.
- Message URLs are useful only to Telegram users who already have access to the source group.

## Failure model

- If the bot exits, the macOS LaunchAgent restarts it.
- A duplicate poller creates a Telegram `409 Conflict`; stop the extra instance rather than retrying indefinitely.
- A transient Telegram API failure during startup can prevent `setMyCommands`; the LaunchAgent restart policy will retry. Check logs if the process is not running.
- Native videos, animations, and video documents are all treated as video media only when Telegram sends a captionable update.
